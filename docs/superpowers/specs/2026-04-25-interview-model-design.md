# Interview Models — Design

**Date:** 2026-04-25
**Status:** Approved
**Scope:** Track 03 next entity. `InterviewCycle` + `InterviewStep`. Backend-only — model + migration + admin + per-app tests. **No DRF API in this PR**, mirroring PR F (Opportunity model) → PR G (Opportunity API) split that worked well.

## Goal

Land the interview-pipeline scaffolding so a future PR can add an API + frontend for "what stages of which company's process am I in". Pure data model — no scheduling, no transcripts, no insights.

## Non-goals

- DRF API (separate later PR).
- Transcripts, recordings, AI summaries (Track 06 territory).
- Calendar integration / reminders.
- State-machine enforcement on `status` transitions (a `CharField(choices=...)` is enough; transition rules land with the API).
- Any link to `BaseResume`, `ResumeVersion`, or other Track 05 entities — that's the next PR after this one.
- Frontend / e2e changes.

## Architecture

### Models (`backend/interviews/models.py`)

```py
class InterviewCycle(TimeStampedModel):
    """One end-to-end loop with a single company for a single opportunity."""
    id = UUIDField(pk, default=uuid4, editable=False)
    opportunity = FK("opportunities.Opportunity", on_delete=CASCADE,
                     related_name="interview_cycles")
    name = CharField(max_length=200)
    started_at = DateTimeField(null=True, blank=True)
    closed_at = DateTimeField(null=True, blank=True)
    notes = TextField(blank=True, default="")

    class Meta:
        ordering = ("-created_at",)


class InterviewStepKind(TextChoices):
    SCREENING = "screening", "Screening"
    PHONE = "phone", "Phone screen"
    TECHNICAL = "technical", "Technical"
    SYSTEM_DESIGN = "system_design", "System design"
    BEHAVIORAL = "behavioral", "Behavioral"
    PANEL = "panel", "Panel"
    OFFER = "offer", "Offer"
    OTHER = "other", "Other"


class InterviewStepStatus(TextChoices):
    SCHEDULED = "scheduled", "Scheduled"
    COMPLETED = "completed", "Completed"
    CANCELLED = "cancelled", "Cancelled"
    NO_SHOW = "no_show", "No show"


class InterviewStep(TimeStampedModel):
    """One step inside a cycle — a phone screen, a system-design loop, …"""
    id = UUIDField(pk, default=uuid4, editable=False)
    cycle = FK(InterviewCycle, on_delete=CASCADE, related_name="steps")
    sequence = PositiveIntegerField()       # 1-indexed; unique per cycle
    kind = CharField(choices=InterviewStepKind.choices, default=OTHER)
    status = CharField(choices=InterviewStepStatus.choices, default=SCHEDULED)
    scheduled_for = DateTimeField(null=True, blank=True)
    duration_minutes = PositiveIntegerField(null=True, blank=True)
    interviewer = CharField(max_length=200, blank=True, default="")
    notes = TextField(blank=True, default="")

    class Meta:
        ordering = ("sequence",)
        constraints = [
            UniqueConstraint(fields=("cycle", "sequence"),
                             name="uniq_interview_step_sequence"),
            CheckConstraint(condition=Q(sequence__gte=1),
                            name="interview_step_sequence_gte_1"),
        ]
```

Choices:

- **UUID pk** — matches Workspace, Opportunity, McpToken, BaseResume.
- **Cycles cascade on opportunity delete.** A cycle without its opportunity is meaningless. Steps cascade on cycle delete for the same reason.
- **`opportunity` FK** instead of `workspace` — the workspace association comes through `cycle.opportunity.workspace`, so we don't duplicate the link. Audit and authz hooks can traverse the path.
- **`name`** is human-supplied (e.g. "Acme — Staff Backend"), not auto-derived. Lets a user log "round 2 with Acme" as a separate cycle from "round 1".
- **`started_at` / `closed_at` are nullable.** A cycle can be "in flight" indefinitely; the closed timestamp is what surfaces in stats later.
- **`InterviewStepKind` covers the common loops** without going overboard. `OTHER` is the escape hatch for anything we don't anticipate.
- **`InterviewStepStatus`** keeps the lifecycle simple: scheduled → completed / cancelled / no_show. State-machine enforcement is deferred.
- **`sequence` is 1-indexed and unique per cycle**, with a `CheckConstraint` enforcing `>= 1` (same shape as `ResumeVersion.version_number`). The API PR will compute the next number; this PR just enforces the invariant.
- **No `is_current` flag**; the latest step is whatever has the highest `sequence`.

### Migration

`backend/interviews/migrations/0001_initial.py`, autogenerated. Two `CreateModel` ops + one `UniqueConstraint` + one `CheckConstraint`. Django auto-indexes both ForeignKeys, so there are no extra `Meta.indexes` to add on top of those.

### Admin (`backend/interviews/admin.py`)

- `InterviewStepInline(TabularInline)` embedded in `InterviewCycle` — read-only stays for `sequence` (so admins can't accidentally renumber), the rest editable.
- `InterviewCycleAdmin`: `list_display=("name","opportunity","started_at","closed_at","created_at")`, `list_filter=("opportunity__workspace",)`, `search_fields=("name","opportunity__title","opportunity__company")`.
- `InterviewStepAdmin`: `list_display=("cycle","sequence","kind","status","scheduled_for","interviewer")`, `list_filter=("kind","status","cycle__opportunity__workspace")`, `search_fields=("interviewer","notes")`.

The `interviews` app is already in `INSTALLED_APPS`. No settings change.

### Tests (per-app per-category layout)

`backend/interviews/tests/models/`:

- `interview_cycle_test.py` — six cases:
  1. Minimum-field create (just `opportunity`, `name`); defaults are set.
  2. `__str__` formats as `f"{name} ({opportunity.title} @ {opportunity.company})"`.
  3. Opportunity cascade.
  4. Default ordering newest-first.
  5. Two cycles for the same opportunity coexist (no unique on `(opportunity, name)` — multiple rounds are valid).
  6. `closed_at` may be set independently and round-trips.

- `interview_step_test.py` — seven cases:
  1. Minimum-field create with `sequence=1`; default `kind=OTHER`, `status=SCHEDULED`.
  2. `__str__` formats as `f"{cycle.name} step {sequence} ({kind})"`.
  3. `(cycle, sequence)` unique — second create with same pair raises `IntegrityError`.
  4. `sequence=0` is rejected by the check constraint.
  5. Two cycles can each have `sequence=1`.
  6. Cascade on cycle delete.
  7. Default ordering inside one cycle is ascending by sequence.

Total +13 tests. Backend pytest: 159 → 172.

## Risk & rollback

Two new tables, no schema change to existing tables. No public surface (admin only, staff + 2FA). No call sites in this PR — the audit-wire pattern from PR #28 will be applied when the API lands.

Rollback: revert the merge. `manage.py migrate interviews zero` drops both tables.

## Open questions

- Whether `closed_at` should auto-derive from "every step is completed/cancelled". Deferred — the API can compute that as a property; we don't need a denormalized column yet.
- Re-ordering steps. The `sequence` UniqueConstraint protects integrity; gaps after a cancelled step are fine. A reorder feature ships with the API, not the model.
